# BUG-021 — Profile MCQs Attempted always 0

**Status:** FIXED  
**Severity:** MEDIUM  
**Category:** data-sync / cross-db  
**Session:** S28  
**Commit:** TBD

## Symptom
`/profile` Study Snapshot shows "0 MCQs ATTEMPTED" regardless of how many RBI MCQ drill sessions the user has completed.

## Root Cause (two-layer)

1. **Wrong table**: `profile_bp.py` queries `return_quiz_attempts` in ies.db. That table holds IES return-quiz MCQs only. RBI MCQ drill attempts are saved to `rbi_attempts` in `rbi.db`. UPSC MCQs live in `upsc.db` → `return_quiz_attempts`.

2. **Wrong connection scope**: `g.rbi_conn` and `g.upsc_conn` are opened by `@before_request` hooks scoped to `rbi_prep_bp` and `upsc_bp` respectively. Those hooks do not fire when `/profile` is requested — so the connections are `None` on that route even if the code were corrected to reference them.

## Fix
`profile_bp.py` now opens short-lived direct connections to `rbi.db` and `upsc.db` within the route handler, queries each for the user's attempt count, and sums all three:

```python
mcqs_attempted = ies_mcqs + rbi_mcqs + upsc_mcqs
```

Pattern: same as the cross-DB two-query Python-merge pattern documented in CLAUDE.md — just applied at the connection level rather than join level.

## Pattern
**SYNC-001**: Any aggregate stat displayed on a page that spans multiple `.db` files must open its own short-lived connections; it cannot rely on blueprint-scoped `g.*_conn` handles that only exist for routes in that specific blueprint.
