# BUG-009: Transaction Rollback Swallows gap_state_events Silently

## Summary
In `submit_return_quiz()`, the `with conn:` transaction block writes to multiple tables:
return_quiz_attempts → gap_states → gap_state_events → user_mastery → topic_attempt_summary.
If an exception occurs after `gap_state_events` is inserted but before the last write, the SQLite
context manager rolls back everything correctly. However, this failure is silent — the user sees
no error, and no event is logged anywhere.

## Impact
- Audit trail has gaps: a gap_state can change without any corresponding event record
- Downstream analytics relying on the event log will miss these transitions
- Hard to debug later why a topic shows unexpected state with no history

## Recommended Fix
Wrap `submit_return_quiz()` in a try/except in the calling page, show a user error on failure,
and log a diagnostic event with the exception details.

## Fix (S26)
`submit()` in `ies_return_quiz_bp.py` now wraps `submit_return_quiz()` in try/except.
On failure: logs to `current_app.logger` with full traceback, sets `session["rq_error"]`,
and redirects back to the quiz page so the user sees a clear error message.

## Priority
Low. The gap_states rollback correctly (state is consistent). Only the audit trail suffers.
