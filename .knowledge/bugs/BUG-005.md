# BUG-005: attempt_count Race Condition — Read-Modify-Write

## Summary
`submit_return_quiz()` reads `attempt_count` from DB in Python, increments it, then writes it back.
Two concurrent submissions (e.g., same user with two browser tabs) both read the same value and both
write the same incremented value — one increment is lost.

## Analogy
Two people both checking a bank balance (₹1000), each adding ₹100 separately, and both writing back
₹1100. The bank ends up with ₹1100 instead of ₹1200 — one deposit disappeared.

## Impact
- attempt_count is off by 1 per concurrent collision
- `stuck_flag` detection (fires when attempt_count >= 3 AND state == FLAGGED) may fire late
- mastery averaging uses `old_count` — same issue, concurrent submissions both use stale count

## Fix Applied
`web/db.py` lines 558, 564, 571: replaced `attempt_count=?` (Python value) with
`attempt_count=attempt_count+1` (SQL expression). The DB engine handles the increment atomically.
The Python `attempt_count` variable is kept for local `stuck_flag` logic only (low-stakes calculation).

## Files Changed
- `web/db.py` lines 558, 564, 571 (three UPDATE statements in submit_return_quiz)
