# BUG-010: set_topic_state() Calls get_user_id() Internally — Wrong-User Writes

## Summary
`db.py:set_topic_state()` resolves the user_id internally via `get_user_id()` instead of accepting
it as a parameter. Same issue with `get_topics()`, `get_attempts()`, `get_attempt_summary()`,
`submit_return_quiz()`, and others.

If `st.session_state.user_id` is stale (e.g., race between two reruns, or bleed from previous user
not fully cleaned), these functions silently write to the wrong user's data.

## Specific Calls on Dashboard.py
Lines 162, 167, 234, 239, 244 — all call `set_topic_state()` without passing user_id, relying on
session_state being correct at call time.

## Recommended Fix
Refactor these functions to accept `user_id` as an explicit parameter. The calling page already has
`user_id` from `require_user()` — pass it through instead of re-resolving internally.

Priority functions to fix first:
- `set_topic_state(conn, topic_id, new_state, trigger)` → add `user_id` param
- `submit_return_quiz(conn, topic_id, answers, session_id)` → already called with user_id available

## Priority
Medium now (BUG-004 session bleed is fixed, reducing exposure). High before multi-device usage.
