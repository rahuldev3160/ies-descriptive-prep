# BUG-022 — Progress page "Time per page" always shows stale/empty data

**Status:** FIXED  
**Severity:** HIGH  
**Category:** data-sync / cross-db / event-type mismatch  
**Session:** S28  
**Commit:** TBD

## Symptom
`/progress` "Today's Activity" and "This Week" bars either show nothing or show 3 stale entries from before the S25 nyaya.db migration. No current session data appears.

## Root Cause (two-layer)

1. **Wrong DB**: `get_time_breakdown(conn, ...)` used the passed `conn` (ies.db). Since S25 all event writes go to nyaya.db via `track_page_time` / `log_event`, both of which call `get_nyaya_conn()` internally and ignore the passed `conn`.

2. **Wrong event_type**: `get_time_breakdown` filtered `event_type='page_time'` but `track_page_time` writes `event_type='page_view'`. These never matched — `page_time` events in the DBs are 3 legacy rows from the pre-S25 era (migrated into nyaya.db but never added to again).

Both failures compound: even reading from the right DB would still return zero rows because the event_type doesn't match.

## Fix
`get_time_breakdown` now calls `get_nyaya_conn()` directly (ignoring the `conn` arg, consistent with how `log_event` / `track_page_time` behave), and filters on `event_type='page_view'` to match what `track_page_time` actually writes.

## Related
- Same pattern as SYNC-001 (BUG-021) — writes and reads split across DBs after S25 migration
- Design smell: `log_event(conn, ...)`, `track_page_time(conn, ...)`, `get_time_breakdown(conn, ...)` all take a `conn` arg that they silently ignore — stale interface from pre-nyaya.db era
