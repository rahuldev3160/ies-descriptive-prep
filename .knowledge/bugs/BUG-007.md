# BUG-007: DB Connection Leaks — 12+ st.stop() Paths Bypass conn.close()

## Summary
Every page opens `conn = get_conn()` at the top and closes it at the bottom. But `st.stop()` is used
throughout pages as an early-exit mechanism. Any `st.stop()` called before `conn.close()` leaks the
connection. SQLite WAL mode holds file locks until all connections release them.

## Affected Paths
| File | st.stop() lines | Description |
|------|----------------|-------------|
| 2_Quiz.py | 147, 180, 313, 440, 474, 486, 524 | No questions, rate limit, API fail, etc. |
| 5_Return_Quiz.py | 66, 98, 184 | No topics, no MCQs, after result |
| 4_My_Progress.py | 99, 119 | No attempts |
| 3_Study_Brief.py | 49 | No topic selected |
| 7_UPSC_Mains.py | 183, 269 | Early exits |
| 1_Model_Answers.py | 339 | Early exit |

## Recommended Fix
Wrap each page's main body in try/finally:
```python
conn = get_conn()
try:
    user_id = require_user(conn)
    # ... page logic ...
finally:
    conn.close()
```
This ensures conn.close() always runs regardless of st.stop() or exceptions.

## Current Impact
Low in practice with SQLite + WAL mode (connections GC'd eventually). Will cause WAL file growth
and `SQLITE_BUSY` errors under high concurrent load. Not critical pre-launch.

## Priority
Fix before scaling to 50+ concurrent users.
