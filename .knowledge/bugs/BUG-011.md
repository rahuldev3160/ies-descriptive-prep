# BUG-011: "rahul" Fallback in get_user_id() — Ghost Data Under Literal String

## Summary
`db.py:23` defines `USER_ID = os.environ.get("IES_USER_ID", "rahul")`. `get_user_id()` falls back
to this if Streamlit is unavailable or raises an exception.

`log_event()` calls `get_user_id()` internally. If Streamlit import fails (edge case), all event
logs get written under user_id="rahul" — a literal string that could accumulate data silently,
contaminating analytics and making it impossible to attribute those records to a real user.

## Scenarios Where This Fires
1. Any page that calls `log_event()` before `require_user()` has set session_state.user_id
2. Script-mode invocations (e.g., running pages directly, not via streamlit) — `IES_USER_ID` env var
   would catch this in dev, but production has no `IES_USER_ID` set, falling back to "rahul"
3. Exception in Streamlit session_state access (rare)

## Recommended Fix
Remove the `USER_ID = "rahul"` default. Replace with: if `get_user_id()` cannot resolve a real UUID,
return `None` and have all callers skip the write rather than write under a phantom ID.

## Priority
Medium. Low probability in production but could silently corrupt analytics.
